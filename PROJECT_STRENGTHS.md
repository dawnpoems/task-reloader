# Task Reloader Portfolio Notes

## 프로젝트 개요

Task Reloader는 반복 작업을 "캘린더의 고정 날짜"가 아니라 "마지막 완료 시점" 기준으로 다시 스케줄링하는 작업 관리 프로젝트다.

일반적인 반복 작업 서비스가 `매주 월요일`, `매달 1일`처럼 날짜 중심으로 반복을 계산한다면, 이 프로젝트는 아래 규칙을 중심으로 설계했다.

```text
next_due_at = completed_at + every_n_days
```

이 방식은 집안일, 점검, 운영성 업무처럼 "언제 실제로 수행했는가"가 중요한 작업에 더 적합하다고 판단했다.

## 이 프로젝트에서 다룬 문제

이 프로젝트에서는 단순 CRUD 구현보다, 반복 작업 도메인에서 자주 발생하는 아래 문제를 중심으로 설계했다.

- 완료 시점 기준으로 다음 일정을 어떻게 계산할 것인가
- 오늘 해야 할 일과 지연된 일을 어떤 기준으로 나눌 것인가
- 완료 요청이 동시에 들어올 때 어떻게 정합성을 유지할 것인가
- 프론트엔드와 백엔드가 서로 다른 상태 판정을 하지 않도록 어떻게 책임을 분리할 것인가

## 핵심 설계

### 1. completion-driven rescheduling

Task의 핵심 상태는 `next_due_at`이다.  
작업이 완료되면 `completed_at`, `last_completed_at`, `next_due_at`를 함께 갱신하도록 설계했다.

이렇게 하면 반복 규칙이 도메인 중심으로 유지되고, 사용자가 실제로 작업을 끝낸 시점을 기준으로 다음 일정이 자연스럽게 밀린다.

### 2. 상태 저장 대신 계산

`OVERDUE`, `TODAY`, `UPCOMING` 상태를 DB 컬럼으로 저장하지 않고, `next_due_at`과 날짜 경계를 비교해서 계산하도록 구현했다.

이 방식을 선택한 이유는 다음과 같다.

- 상태와 실제 일정이 어긋나는 문제를 줄이기 위해
- 저장해야 하는 값을 최소화하기 위해
- 조회 시점의 상태를 항상 같은 규칙으로 계산하기 위해

### 3. 시간 경계 명시

기준 타임존은 `Asia/Seoul`로 두고, 상태는 아래 기준으로 분류했다.

- `OVERDUE`: `next_due_at < 오늘 00:00`
- `TODAY`: `오늘 00:00 <= next_due_at < 내일 00:00`
- `UPCOMING`: `next_due_at >= 내일 00:00`

날짜 경계를 별도 객체로 분리해서 사용했고, 경계값 테스트도 함께 작성했다.

### 4. 완료 처리의 동시성 대응

완료 API에는 두 가지 방어 장치를 넣었다.

- DB row lock
- 2초 이내 중복 완료 요청 차단

row lock은 동시에 같은 Task를 갱신하는 상황에서 데이터 정합성을 지키기 위한 목적이고, 중복 완료 차단은 더블클릭이나 연속 요청 같은 사용자 입력 상황을 방어하기 위한 목적이다.

## 기술 선택과 이유

### Backend

- Java 17
- Spring Boot
- Spring Data JPA
- Flyway
- PostgreSQL

Spring Boot와 JPA를 사용해 도메인 규칙, 트랜잭션, DB 접근을 명확하게 나누고자 했다.  
Flyway를 사용해 스키마 변경 이력을 코드와 함께 관리하도록 구성했다.

### Frontend

- React
- Vite
- TypeScript

프론트엔드는 복잡한 상태관리 도구 없이 `useTasks` 훅에 API 호출, 에러 처리, 토스트, 목록 갱신 흐름을 모아 현재 규모에 맞는 단순한 구조를 유지했다.

### Infra

- Docker
- Docker Compose
- nginx

로컬 개발과 컨테이너 기반 실행을 모두 지원하도록 구성했다.  
개발 환경에서는 Vite proxy를 사용하고, Docker 환경에서는 nginx가 정적 파일과 API 프록시를 함께 담당한다.

## 구현하면서 신경 쓴 부분

### 도메인 규칙이 분산되지 않도록 구조화

핵심 일정 갱신 규칙은 엔티티와 서비스 계층에 모아두고, 프론트엔드가 상태를 다시 계산하지 않도록 했다.  
비즈니스 규칙의 기준점을 백엔드에 두어 프론트와 백엔드의 판정 불일치를 줄이려는 의도였다.

### 날짜/시간 관련 버그 가능성 축소

반복 작업은 날짜 경계에서 오류가 나기 쉬워서, `오늘`, `내일`, `시작일`, `완료 시점`을 명확한 시간 값으로 다루는 데 집중했다.

### 작은 규모에 맞는 단순한 구조 유지

기능 수가 많지 않은 프로젝트이기 때문에 과한 추상화나 불필요한 라이브러리 도입을 피했다.  
대신 도메인 규칙, 테스트, 실행 구조를 비교적 명확하게 보여주는 데 집중했다.

## 테스트 범위

다음 범위를 중심으로 테스트를 구성했다.

- 상태 판정 경계값 테스트
- 서비스 레이어 비즈니스 규칙 테스트
- 컨트롤러 요청/응답 테스트
- JPA repository 테스트
- `startDate`와 `nextDueAt` 초기화 규칙 테스트

기능 수를 넓게 덮기보다, 날짜 계산과 완료 처리처럼 실수하기 쉬운 핵심 규칙을 우선 검증하는 방향으로 잡았다.

## 프로젝트 구조

```text
task-reloader/
├── apps/
│   ├── api/      # Spring Boot API
│   └── web/      # React frontend
├── infra/        # Docker Compose, 환경 설정
└── README.md
```

## 확장 가능성

현재는 MVP 범위로 구현했지만, 아래 방향으로 자연스럽게 확장할 수 있게 구조를 잡았다.

- 사용자 계정 및 다중 사용자 지원
- 반복 단위 확장: weekly, monthly
- 알림 기능
- 완료 이력 로그
- 대시보드 및 통계

핵심 도메인 규칙과 시간 계산 로직이 분리되어 있어, 이후 기능을 추가할 때도 중심 개념을 유지하기 쉬운 구조다.

## 한 줄 정리

Task Reloader는 반복 작업을 완료 시점 기준으로 다시 계산하는 방식을 중심에 둔 프로젝트이며, 시간 경계 처리, 상태 계산, 완료 API의 동시성 대응을 핵심 구현 포인트로 삼아 설계했다.
